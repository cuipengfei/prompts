# 优化提示词

> **铁律**：先还原用户真实意图，再结构化改写。补全逻辑，不编造事实；减少误解，不扩大范围。

## 核心边界

**允许**：
- 重组表达，让目标、上下文、步骤、输出和约束更清晰。
- 将用户已暗示、任务逻辑必需的信息显式化，并标为“推断”。
- 补充 LLM 完成任务所需的操作性要求，如格式、验证、边界、成功标准。
- 补充可验证 Definition of Done：把“做好、完整、高质量”等模糊目标转成可观察完成条件、证据来源和验证动作。
- 在信息不足会改变方向时，先问 1 个最关键澄清问题。

**禁止**：
- 添加用户未表达、未暗示、也非任务必需的新需求。
- 把猜测写成事实，或替用户决定业务偏好。
- 删除用户明确要求，或改变任务目标、技术栈、范围、语气偏好。
- 把未授权的执行动作写进 Definition of Done；若用户只要求提案、分析或改写，DoD 只能验证该交付物本身，不能要求实际改文件、部署或外部操作。
- 为了显得完整而加入模板化废话。

## 执行流程

### 1. 意图诊断

先分析原始提示词，不急着改写：

| 项 | 判断问题 |
|----|----------|
| 真实目标 | 用户最终想完成什么，而不只是字面动作？ |
| 成功标准 | 什么结果会让用户认为任务做对了？ |
| 可验证 DoD | 哪些条件满足后可判定任务完成？每条是否有可观察证据？ |
| 验证方式 | 用阅读、命令、测试、引用、截图、人工检查，还是明确标注未验证？ |
| 已知约束 | 用户明确给了哪些范围、禁用项、格式、工具或风格？ |
| 缺失上下文 | 哪些信息缺失会导致 LLM 走错方向？ |
| 可合理推断 | 哪些信息可由上下文推出，但必须标明是推断？ |
| 高风险歧义 | 哪些歧义一旦猜错会明显改变方案？ |

### 2. 澄清或继续

- 若高风险歧义存在：只问 1 个最关键问题，并说明为什么这个答案会改变执行方向。
- 若缺失信息不影响主方向：继续改写，在提示词中加入“未指定时采用保守默认”的约束。
- 若只是表达松散：直接结构化改写。

常见高风险歧义：目标受众、交付物类型、技术栈、数据来源、时间范围、权限边界、是否允许联网、是否要执行还是只分析。

### 3. 结构化改写

按任务需要裁剪以下结构，不必机械保留所有标题：

```markdown
## 目标
[一句话说明用户真实目标]

## 背景与上下文
- 已知：[用户明确提供的信息]
- 推断：[从上下文合理推出的信息]
- 未指定：[会影响细节但不阻塞执行的信息]

## 执行要求
1. [步骤]
2. [步骤]

## 输出要求
- [格式、粒度、语言、长度、证据要求]

## Definition of Done
任务完成必须同时满足：
1. [完成条件]
   - 证据：[文件、命令输出、测试结果、引用来源、截图、清单项或人工可检查信号]
   - 验证方式：[如何确认]
   - 失败处理：[不满足时怎么报告、继续或标注未验证]
2. [完成条件]
   - 证据：[...]
   - 验证方式：(...)
   - 失败处理：(...)

## 约束
- 必须：[用户明确要求或任务必要约束]
- 禁止：[用户明确禁止或为防止跑偏必须限制的行为]

## 非目标
- [明确不做什么，避免 DoD 扩范围]

## 对齐检查
- 若遇到 [关键不确定点]，先提问；否则按 [保守默认] 执行。
```

#### Definition of Done 生成规则

- 每条 DoD 必须可观察：避免只写“高质量、完善、合理、足够好”等无法直接验证的形容词。
- 每条 DoD 必须绑定证据：文件、命令输出、测试结果、引用来源、清单项或明确人工检查点。
- 区分 DoD 与验收条件：通用质量门槛放 DoD；单个功能、内容或故事的具体满足条件放执行要求或验收条件。
- 保持最小足够：只写判断完成必需的条件，不穷举所有理想改进。
- 不越权：用户说“propose only”“只分析”“不要执行”时，DoD 只验证提案或分析质量，不加入实际修改、部署、发送、安装等动作。

### 4. 对齐验证

输出前逐项检查：

- 原始每个明确要求都保留。
- 每个新增内容都属于：结构化表达、任务必需步骤、合理推断、或防止跑偏的边界。
- 每条 DoD 都有可观察证据和验证方式；无法验证的部分已标为“未验证”或“需人工检查”。
- 推断和未知被明确标注，没有伪装成事实。
- 改写后比原文更容易执行，且没有改变用户目标。

## 输出格式

```markdown
### 原始提示词
[用户原文]

---

### 增强版提示词
[改写后的版本]

---

### 改进说明
- 真实意图：[识别到的核心目标]
- 缺口处理：[补了哪些必要上下文；哪些仍标为未指定]
- 推断边界：[哪些内容是推断，为什么合理]
- 防跑偏约束：[新增了哪些边界以避免错误方向]
- DoD 设计：[新增了哪些完成条件；它们如何被验证]
- 验证边界：[哪些条件可自动验证；哪些只能人工检查；哪些当前未指定]
- 保持不变：[确认未扩展范围]
```

若需要追问，先输出：

```markdown
### 需要澄清
[一个关键问题]

### 原因
[为什么不问会导致方向跑偏]
```

## 工具推荐

仅当工具与用户原始目标直接相关，且不会扩大范围时，推荐 1-3 个当前可用工具。无强相关工具时省略。

格式：`工具名` - 一句话说明用途。

## 执行增强版提示词

完成改写后，立即执行增强版提示词，除非：

- 用户明确只要求改写，不要求执行。
- 改写阶段触发了必须澄清的问题。
- 执行需要用户未授权的外部操作。

执行时使用增强版提示词的范围和约束，不再自行新增目标、工具、技术栈或交付物。
